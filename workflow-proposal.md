# مقترح Workflow الجديد — نظام HBOperations

## الفكرة الأساسية

إضافة **إدارة الشؤون الإدارية** كوسيط لوجستي (مناولة/توصيل) بين المرسل والمستلم.
هذا يُحاكي الواقع الفعلي في البنك: المستندات لا تنتقل مباشرة، بل عبر قسم متخصص يتتبع حركتها.

## الحالات (States)

| # | الحالة | الاسم العربي | الوصف | من يراها |
|---|--------|-------------|-------|----------|
| 0 | `Sent` | مُرسلة | المرسل أنشأ المعاملة، بانتظار الشؤون الإدارية | المرسل + الشؤون الإدارية |
| 1 | `InTransit` | قيد التوصيل | الشؤون الإدارية استلمت من المرسل وأكدت التوجيه للمستلم | الشؤون + المستلم |
| 2 | `Received` | مُستلمة | المستلم أكّد الاستلام — المعاملة مكتملة | الجميع |
| 3 | `Rejected` | مرفوضة | رُفضت مع سبب (من الشؤون أو المستلم) | الجميع |
| 4 | `Cancelled` | ملغاة | المرسل ألغى قبل استلام الشؤون | المرسل فقط |
| 5 | `Archived` | مؤرشفة | أُرشفت تلقائياً أو يدوياً | الجميع |

## الانتقالات المسموحة (Transitions)

| من | إلى | من ينفّذ | الشروط | ماذا يحدث |
|----|-----|----------|--------|-----------|
| `Sent` | `InTransit` | موظف الشؤون الإدارية | — | يسجّل وقت الاستلام من المرسل + ملاحظة اختيارية |
| `Sent` | `Rejected` | موظف الشؤون الإدارية | سبب إجباري | مستندات ناقصة/تالفة/غير مطابقة |
| `Sent` | `Cancelled` | المرسل نفسه | المعاملة ما زالت Sent | المرسل يلغي قبل أن تستلمها الشؤون |
| `InTransit` | `Received` | المستلم المحدد | وقت الاستلام ≥ وقت الإرسال | يسجّل وقت الاستلام + مستند إثبات (إن طُلب) |
| `InTransit` | `Rejected` | المستلم المحدد | سبب إجباري | المستلم يرفض (ليست له/خطأ في التوجيه) |
| `Received` | `Archived` | النظام / Admin | — | أرشفة تلقائية أو يدوية |
| `Rejected` | `Archived` | النظام / Admin | — | أرشفة تلقائية أو يدوية |

## الأدوار في الـ Workflow

### 1. المرسل (Sender)
- **من هو**: أي موظف في أي فرع/إدارة
- **يفعل**: إنشاء معاملة + إلغاء (قبل استلام الشؤون فقط)
- **يرى**: معاملاته المُرسلة + حالتها الحالية

### 2. الشؤون الإدارية (Admin Affairs) — الدور الجديد
- **من هم**: موظفو إدارة الشؤون الإدارية (DepartmentManager + DepartmentStaff في إدارة الشؤون)
- **يفعلون**:
  - استلام المعاملات من المرسلين (Sent → InTransit)
  - رفض المعاملات الناقصة/التالفة (Sent → Rejected)
- **يرون**: كل المعاملات في حالة `Sent` (قائمة الانتظار) + المعاملات `InTransit` التي أرسلوها

### 3. المستلم (Receiver)
- **من هو**: الشخص المحدد بالاسم في المعاملة
- **يفعل**: تأكيد الاستلام + رفض
- **يرى**: المعاملات الموجهة إليه في حالة `InTransit`

### 4. المشرفون (Supervisory)
- **SuperAdmin/ITAdmin**: أرشفة يدوية + تعديل إداري
- **CEO/Auditor/Compliance/Shariah**: اطلاع فقط على كل المعاملات

| الحقل | متى يُسجَّل | من يسجّله |
|-------|------------|-----------|
| `SentAt` | عند إنشاء المعاملة | المرسل (يدخل الوقت الفعلي للإرسال) |

| `ReceivedByUserId` | عند انتقال InTransit → Received | تلقائي (الموظف المسجّل) |
| `CompletedAt` | عند Received أو Rejected | تلقائي |

## سيناريوهات عملية

### السيناريو 1: مسار ناجح (الأغلب)
```
1. عوض (إدارة IT) يُنشئ معاملة "تسليم مستندات" لأيمن (فرع فوه)
2. المعاملة تظهر في قائمة الشؤون الإدارية ← حالة: Sent
3. سامح (مدير الشؤون) أو موظفه يؤكد الاستلام ← حالة: InTransit
   - يُسجَّل: PickedUpAt + PickedUpByUserId
4. أيمن يرى المعاملة في قائمته ← يؤكد الاستلام ← حالة: Received
   - يُسجَّل: ReceivedAt + ReceivedByUserId + مستند إثبات
5. بعد 30 يوم ← أرشفة تلقائية ← حالة: Archived
```

### السيناريو 2: الشؤون ترفض (مستندات ناقصة)
```
1. موظف يُنشئ معاملة
2. الشؤون تفحص وتجد مستندات ناقصة ← ترفض مع سبب ← حالة: Rejected
3. المرسل يتلقى إشعار بالرفض + السبب
4. المرسل يُنشئ معاملة جديدة بعد استكمال المستندات
```

### السيناريو 3: المستلم يرفض
```
1. معاملة في InTransit
2. المستلم يرى أن المعاملة ليست له / خطأ في التوجيه ← يرفض مع سبب
3. حالة: Rejected
4. المرسل يتلقى إشعار
```

### السيناريو 4: المرسل يلغي
```
1. المرسل أنشأ معاملة بالخطأ
2. الشؤون لم تستلمها بعد (ما زالت Sent)
3. المرسل يلغي ← حالة: Cancelled (أو حذف فعلي)
```

|--------|-----------|-----------|------------|

| `Received` | اطلاع | اطلاع | اطلاع |
| `Rejected` | اطلاع + سبب الرفض | اطلاع | اطلاع |

| البند | القديم | الجديد |
|-------|--------|--------|
| عدد الحالات | 5 (Sent, Received, Rejected, Cancelled, Archived) | 6 (+InTransit) |

| الرؤية | المستلم يرى فوراً | المستلم يرى بعد تأكيد الشؤون |

## القرارات التصميمية

### 1. هل الإلغاء يحذف أم يغيّر حالة؟
**التوصية**: تغيير حالة إلى `Cancelled` (لا حذف) — للحفاظ على سجل كامل ومراجعة لاحقة.

### 2. هل المستلم يرى المعاملة وهي Sent؟
**التوصية**: **لا** — المستلم يرى فقط بعد `InTransit`. هذا يمنع الارتباك ويعكس الواقع (المستندات لم تصله بعد).

### 3. هل يمكن للشؤون تعديل المستلم؟
**التوصية**: **لا** — الشؤون إما تقبل أو ترفض. إذا التوجيه خاطئ، ترفض ويُنشئ المرسل معاملة جديدة.

### 4. هل كل المعاملات تمر عبر الشؤون؟
**التوصية**: **نعم** — لتوحيد المسار. إذا أردنا استثناء (مثلاً: داخل نفس الإدارة)، يمكن إضافة `BypassAdmin` flag لاحقاً.

### 5. من يمثّل "الشؤون الإدارية" تقنياً؟
**التوصية**: أي مستخدم له دور `DepartmentManager` أو `DepartmentStaff` في **إدارة الشؤون الإدارية** تحديداً. يُحدد بـ DepartmentId.
